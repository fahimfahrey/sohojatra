# Optional TOTP Two-Factor Authentication — Design

**Date:** 2026-05-23
**Status:** Approved (autonomous mode)

## Goal

Let a user opt into a second authentication factor — a TOTP code generated by an authenticator app (Authy, Google Authenticator, 1Password, etc.) — that gates both login and a set of named sensitive operations. A stolen email/password alone must not be sufficient to take over an enrolled account or to perform a sensitive write on its behalf.

Today the only login factor is a password (or Google OAuth, which delegates the second factor to Google). For password-only accounts there is no defense against credential stuffing or post-leak takeover beyond rate limits.

## Background

**Existing auth surface (see `src/app/actions/auth.ts`, `src/lib/supabase/middleware.ts`):**

- Auth provider is Supabase. Sessions are JWT cookies, refreshed per request through `@supabase/ssr`. Server actions read the session via `createClient()` + `supabase.auth.getUser()`.
- Server-action pattern: `"use server"`, `_prev` + `FormData`, returns `ActionResult = { success: true, data? } | { success: false, error }`. CSRF via `validateCsrfToken`. Rate limit via `checkRateLimit(key, max, windowMs)`. Audit via `logAuditEvent`.
- Middleware at `src/middleware.ts` + `src/lib/supabase/middleware.ts` already handles session refresh, idle timeout, CSRF cookie issuance, and the "email-not-confirmed → /email-confirmation" redirect. A "2FA challenge required → /2fa/challenge" redirect fits the same shape.
- Sensitive operations exist as discrete server actions: `createRideAction` (ride creation), `getCreatorPhoneAction` (phone access). The user-facing "phone changes" referenced in the task lives implicitly in `createRideAction` and `joinRideAction` (each accepts a fresh `contactPhone`); there is no standalone "update profile phone" action yet, so step-up applies to those two writes where the phone field is set.

**At-rest encryption precedent (see `SUPABASE_PHONE_ENCRYPTION.sql`):**

- pgcrypto's `pgp_sym_encrypt`/`pgp_sym_decrypt` for symmetric encryption.
- Supabase Vault holds the symmetric key; a `private.<key>_encryption_key()` accessor (SECURITY DEFINER, `REVOKE ALL` from anon/authenticated) is the only path to the plaintext key.
- Encrypted column lives next to the plaintext column type as `bytea`.
- The TOTP feature reuses this pattern: a new vault secret `totp_encryption_key`, a new `private.totp_encryption_key()` accessor, and new bytea columns.

**Why not `supabase.auth.mfa` (Supabase's built-in MFA factor API).** It would deliver TOTP without our writing any crypto. We are not using it because: (a) the task explicitly requires the secret to live in our `users` table encrypted with our key — Supabase MFA stores factors in `auth.mfa_factors`, which our backups, audit log, and key rotation procedures do not cover; (b) Supabase MFA's verification flow issues an `aal2` JWT that our middleware would still need custom code to gate against, so the integration work is not eliminated, only relocated; (c) sensitive-operation step-up ("re-prove 2FA to create a ride") is not a first-class Supabase MFA primitive — we would still build a step-up cookie. Net: equal complexity, less control over secret material, harder to audit. Roll our own using a vetted library.

**Why `otpauth` (npm), not `speakeasy`.** `speakeasy` is last published 2017 and unmaintained. `otpauth` (active, no native deps, edge-runtime safe, ~5 kB) handles secret generation, otpauth:// URI building, and code generation. We use it **only** during enrollment (generate secret app-side, build the otpauth URI for the QR code). All runtime verifications happen in the database via pgcrypto's `hmac()` so the plaintext secret never crosses the Postgres boundary at verify time.

## Approach

Enrollment, verification, and step-up are three small server actions backed by four SECURITY DEFINER RPCs and one helper schema (`private`, already exists). The encrypted secret and the JWT `app_metadata` flag together drive middleware redirects and step-up gates.

1. **Enrollment** — `startTotpEnrollmentAction()` generates a random 20-byte secret app-side, calls `set_pending_totp_secret(p_secret)` RPC which encrypts and stores it in `users.totp_secret_pending_encrypted`, and returns the otpauth URI so the client can render a QR code. `confirmTotpEnrollmentAction(code)` calls `verify_totp_code(p_code, p_use_pending => true)` RPC; on success the RPC promotes the pending secret to `users.totp_secret_encrypted`, sets `totp_enabled = true`, issues 10 single-use recovery codes (returned to the client once, then SHA-256-hashed and stored in `user_totp_recovery_codes`), and stamps `app_metadata.totp_enabled = true` on the Supabase user (server-action calls `supabase.auth.admin.updateUserById` via the service-role client).

2. **Login challenge** — On password sign-in (no change to the action itself), middleware sees `app_metadata.totp_enabled === true` and `!cookie('sb-2fa-passed')`. Redirects to `/2fa/challenge` (allowlisted). `submitTotpChallengeAction(code)` calls `verify_totp_code(p_code, p_use_pending => false)`; on success sets a HMAC-signed `sb-2fa-passed` cookie (TTL = session-bound, `httpOnly`, `secure`, `sameSite=strict`). Recovery code branch: `submitTotpRecoveryAction(code)` matches against `user_totp_recovery_codes`, marks used, sets the same cookie, audits.

3. **Step-up for sensitive ops** — `createRideAction` and `joinRideAction` call `requireFreshTotp()` helper before mutating. If user has `totp_enabled` and the `sb-2fa-stepup` cookie is missing or expired (>15 min), action returns `{ success: false, error: "2FA verification required", code: "2FA_STEPUP_REQUIRED" }`. Client routes to `/2fa/stepup?next=<origin>` which re-uses the challenge form. On success a fresh `sb-2fa-stepup` cookie is issued (HMAC-signed, TTL 15 min).

4. **Disable** — `disableTotpAction(currentPassword, code)` requires password re-prompt AND a valid TOTP/recovery code, then clears `totp_secret_encrypted`, `totp_enabled = false`, deletes recovery codes, clears `app_metadata.totp_enabled`. Audited heavily.

The TOTP secret never leaves Postgres at verify time. The plaintext key never leaves Postgres at all (Vault holds it). The plaintext secret leaves Postgres only at enrollment, in the otpauth URI returned to the client browser exactly once.

**Server actions (all in `src/app/actions/totp.ts`):**

| Action | Returns | Requires |
|---|---|---|
| `startTotpEnrollmentAction(csrf)` | `{ otpauthUri, secretFormatted }` | active session, `!totp_enabled`, CSRF, rate limit `totp:enroll_start:<uid>` 5/hour |
| `confirmTotpEnrollmentAction(code, csrf)` | `{ recoveryCodes: string[] }` | active session, `totp_secret_pending_encrypted` present, CSRF, rate limit `totp:enroll_confirm:<uid>` 10/15min |
| `submitTotpChallengeAction(code, csrf)` | `{ success: true }` then `redirect(safeNext)` | active session, `totp_enabled`, CSRF, rate limit `totp:challenge:<uid>` 5/15min |
| `submitTotpRecoveryAction(code, csrf)` | `{ success: true }` then `redirect(safeNext)` | active session, `totp_enabled`, CSRF, rate limit `totp:recovery:<uid>` 5/hour |
| `submitTotpStepUpAction(code, csrf, next)` | `{ success: true }` then `redirect(next)` | active session, `totp_enabled`, CSRF, rate limit `totp:stepup:<uid>` 10/15min |
| `disableTotpAction(password, code, csrf)` | `{ success: true }` | active session, `totp_enabled`, CSRF, rate limit `totp:disable:<uid>` 3/hour |

**Database surface (all in `SUPABASE_TOTP_2FA.sql`):**

| Function | SECURITY | Purpose |
|---|---|---|
| `private.totp_encryption_key()` | DEFINER, REVOKE from anon/authenticated | Vault accessor |
| `private.totp_compute(p_secret bytea, p_counter bigint) → integer` | INVOKER, REVOKE from anon/authenticated | RFC 6238 truncation. `extensions.hmac(counter_be, p_secret, 'sha1')` → 6-digit int. Pure function. |
| `public.set_pending_totp_secret(p_secret text) → void` | DEFINER, GRANT authenticated | Encrypts `p_secret` (base32) into `users.totp_secret_pending_encrypted` for `auth.uid()` |
| `public.verify_totp_code(p_code text, p_use_pending boolean) → boolean` | DEFINER, GRANT authenticated | Decrypts secret (pending or active), checks code against current and ±1 window. If `p_use_pending`, promotes on success and clears pending. Returns `false` on no match. Logs `auth.totp.verify` failure inside the function (so failures get logged even if app crashes). |
| `public.set_totp_recovery_codes(p_hashes text[]) → void` | DEFINER, GRANT authenticated | Replaces rows in `user_totp_recovery_codes` for `auth.uid()` |
| `public.consume_totp_recovery_code(p_hash text) → boolean` | DEFINER, GRANT authenticated | Atomic `UPDATE … WHERE used_at IS NULL RETURNING 1` |
| `public.disable_totp() → void` | DEFINER, GRANT authenticated | Clears `totp_*` columns and recovery rows |

The `verify_totp_code` RPC is the one place plaintext secrets exist outside encrypted storage, and only inside a SECURITY DEFINER function body — never returned to the client.

## Data Flow

**Enrollment:**
```
User clicks "Enable 2FA" on settings page
  → startTotpEnrollmentAction
     → CSRF, requireUser, rate limit
     → secret = randomBytes(20), base32 encode
     → uri = otpauth://totp/Sohojatra:<email>?secret=<b32>&issuer=Sohojatra&algorithm=SHA1&digits=6&period=30
     → rpc('set_pending_totp_secret', { p_secret: b32 })
     → return { otpauthUri: uri, secretFormatted: humanGrouped(b32) }
  → /2fa/enroll renders QR(uri) + manual code
  → User scans, enters 6-digit code
  → confirmTotpEnrollmentAction(code)
     → CSRF, requireUser, rate limit, totpCodeSchema.parse
     → rpc('verify_totp_code', { p_code: code, p_use_pending: true })
        → on true: promotes pending → active, sets totp_enabled = true
     → if false: audit failure, return error
     → generate 10 codes: 8-char alphanum (40 bits entropy each)
     → hashes = codes.map(c => sha256(c))
     → rpc('set_totp_recovery_codes', { p_hashes: hashes })
     → service-role client: supabase.auth.admin.updateUserById(uid, { app_metadata: { totp_enabled: true } })
     → audit auth.totp.enroll_complete
     → return { recoveryCodes: codes }     -- plaintext, ONE TIME ONLY
  → /2fa/recovery-codes displays + download
```

**Login challenge:**
```
User submits password → signInAction → session set
  → next request: middleware reads JWT
     → if user.app_metadata.totp_enabled && !signedCookie('sb-2fa-passed')
        → redirect /2fa/challenge?next=<original>
  → User enters code
  → submitTotpChallengeAction(code)
     → CSRF, requireUser, rate limit, totpCodeSchema.parse
     → rpc('verify_totp_code', { p_code: code, p_use_pending: false })
     → on true: set signed cookie sb-2fa-passed (httpOnly, secure, sameSite=strict,
               session-only — no Max-Age; cleared on browser close)
     → audit auth.totp.verify success
     → redirect safeNext
```

**Step-up on sensitive write:**
```
User triggers createRideAction
  → requireUser, requireFreshTotp(supabaseUser)
     → if !user.app_metadata.totp_enabled: pass through (no 2FA enabled)
     → if signedCookie('sb-2fa-stepup') valid + exp > now: pass through
     → else: return { success: false, error: "2FA verification required",
                      code: "2FA_STEPUP_REQUIRED" }
  → client sees code === "2FA_STEPUP_REQUIRED", routes to /2fa/stepup?next=/create-ride
  → submitTotpStepUpAction(code, next)
     → CSRF, requireUser, rate limit, totpCodeSchema.parse
     → rpc('verify_totp_code', ...)
     → on true: set signed cookie sb-2fa-stepup (TTL 900s, httpOnly, secure, sameSite=strict)
     → audit auth.totp.stepup success
     → redirect safeNext     -- user resubmits ride form
```

**Disable:**
```
User → settings → "Disable 2FA"
  → disableTotpAction(password, code)
     → CSRF, requireUser, rate limit
     → supabase.auth.signInWithPassword({ email: user.email, password })  -- re-prompt
        → on failure: audit, return error
     → rpc('verify_totp_code', { p_code: code, p_use_pending: false })
     → if false: audit auth.totp.disable failure, return error
     → rpc('disable_totp')        -- clears everything
     → service-role: app_metadata.totp_enabled = false
     → clear sb-2fa-passed and sb-2fa-stepup cookies
     → audit auth.totp.disable success
     → return { success: true }
```

## Database Changes

```sql
-- users table additions
ALTER TABLE public.users
  ADD COLUMN totp_secret_encrypted         bytea,
  ADD COLUMN totp_secret_pending_encrypted bytea,
  ADD COLUMN totp_enabled                  boolean NOT NULL DEFAULT false,
  ADD COLUMN totp_enabled_at               timestamptz,
  ADD COLUMN totp_last_verified_at         timestamptz;

-- recovery codes (one row per code; rows deleted on disable)
CREATE TABLE public.user_totp_recovery_codes (
  user_id    uuid        NOT NULL REFERENCES public.users(id) ON DELETE CASCADE,
  code_hash  text        NOT NULL,                 -- SHA-256 hex, 64 chars
  used_at    timestamptz,
  created_at timestamptz NOT NULL DEFAULT now(),
  PRIMARY KEY (user_id, code_hash)
);

ALTER TABLE public.user_totp_recovery_codes ENABLE ROW LEVEL SECURITY;
ALTER TABLE public.user_totp_recovery_codes FORCE  ROW LEVEL SECURITY;
-- No policies for anon/authenticated: deny all direct access.
-- All access goes through SECURITY DEFINER RPCs above.

-- Vault secret (run ONCE)
DO $$
BEGIN
  IF NOT EXISTS (SELECT 1 FROM vault.secrets WHERE name = 'totp_encryption_key') THEN
    PERFORM vault.create_secret(
      'REPLACE_WITH_STRONG_RANDOM_KEY',     -- openssl rand -base64 48
      'totp_encryption_key',
      'Symmetric key for pgp_sym_encrypt on TOTP secrets'
    );
  END IF;
END $$;
```

Indexes: none beyond the PK on `user_totp_recovery_codes`. The hot path is verify, which does a single `users` row read by primary key.

RLS on `users`: existing `users_select_own` already restricts to `auth.uid() = id`. The new `totp_*` columns are protected by that policy; clients reading their own row can see whether `totp_enabled = true` but cannot read either ciphertext column meaningfully. Authenticated SELECT of `totp_secret_encrypted` returns bytea opaque; no decrypt path exposed.

## Cookies

Two HMAC-signed cookies. Both signed with `process.env.TOTP_COOKIE_SECRET` (new env var, fail fast if absent in production).

| Cookie | TTL | Lifetime | Purpose |
|---|---|---|---|
| `sb-2fa-passed` | session | until browser close OR Supabase session ends | Skip `/2fa/challenge` redirect on login flow |
| `sb-2fa-stepup` | 900s (15 min) | until expiry | Allow sensitive operations without re-prompt |

Both: `httpOnly`, `secure` (in production), `sameSite=strict`, `path=/`. Value format: `<base64url(userId)>.<base64url(expiresAtIso)>.<base64url(hmacSha256(userId+expiresAtIso, secret))>`. Verifier parses, checks expiry, recomputes HMAC, constant-time compares. Cookie tied to `userId` so it cannot be replayed across accounts.

On `signOutAction`: explicit `response.cookies.delete('sb-2fa-passed')` and `delete('sb-2fa-stepup')`.

## Middleware Changes

In `src/lib/supabase/middleware.ts`, after the existing email-confirmation branch:

```ts
const is2faRoute = pathname.startsWith("/2fa");
if (
  user &&
  user.app_metadata?.totp_enabled === true &&
  !is2faRoute &&
  isProtected &&
  !verifySigned2faPassed(request.cookies.get("sb-2fa-passed")?.value, user.id)
) {
  const url = request.nextUrl.clone();
  url.pathname = "/2fa/challenge";
  url.searchParams.set("next", pathname);
  return { response: NextResponse.redirect(url), userId: user.id };
}
```

`isProtected` already covers `/dashboard`, `/create-ride`, `/rides`, `/email-confirmation`. Adding `/2fa/*` to the allowlist (auth-route-ish) so already-logged-in users are not bounced away from the challenge page.

The flag lives in JWT `app_metadata`, so this check is one base64 decode — no per-request DB hit.

## Step-Up Helper

```ts
// src/lib/auth/require-fresh-totp.ts
export async function requireFreshTotp(): Promise<
  { ok: true } | { ok: false; reason: "stepup_required" }
> {
  const user = await requireUser();
  if (user.app_metadata?.totp_enabled !== true) return { ok: true };

  const cookieStore = await cookies();
  const raw = cookieStore.get("sb-2fa-stepup")?.value;
  if (!raw) return { ok: false, reason: "stepup_required" };
  if (!verifySigned2faStepup(raw, user.id)) return { ok: false, reason: "stepup_required" };

  return { ok: true };
}
```

Wired into `createRideAction` and `joinRideAction` right after `requireUser()`. Returns the sentinel error code `2FA_STEPUP_REQUIRED` for the client to detect; client routes to `/2fa/stepup?next=<form-page>`.

## Rate Limits

| Key | Limit | Window | Purpose |
|---|---|---|---|
| `totp:enroll_start:<uid>` | 5 | 1 hour | Bound enrollment churn |
| `totp:enroll_confirm:<uid>` | 10 | 15 min | Bound brute force on initial enrollment code |
| `totp:challenge:<uid>` | 5 | 15 min | Bound brute force on login challenge |
| `totp:stepup:<uid>` | 10 | 15 min | Bound brute force on step-up code (separate from challenge so step-up failures don't lock the user out of login) |
| `totp:recovery:<uid>` | 5 | 1 hour | Bound recovery-code guessing (already very high entropy, kept low to flag misuse) |
| `totp:disable:<uid>` | 3 | 1 hour | Bound disable attempts |

All limits exhausted → `{ success: false, error: "Too many attempts. Try again later." }`, audit with `reason: "rate_limited"`. Critical: rate-limit failures are audited but **do not** count toward the user's standing brute-force lockout — they just delay.

6 digits = 1M codes. With 5 attempts per 15min, expected time to guess = 1M × 15min / 5 ≈ 5.7 years. Combined with the ±1 step window (3 acceptable codes per attempt), expected time ≈ 1.9 years. Tolerable.

## Audit Events

Extend `AuditAction` union in `src/lib/audit.ts`:

```
| "auth.totp.enroll_start"
| "auth.totp.enroll_complete"
| "auth.totp.enroll_abandoned"   -- pending secret cleared without confirm
| "auth.totp.disable"
| "auth.totp.verify"             -- challenge or step-up; detail.kind distinguishes
| "auth.totp.recovery_use"
| "auth.totp.stepup"
```

Detail payloads never include the code, the secret, or the recovery-code plaintext. They include `kind: "challenge" | "stepup"`, `reason: "invalid_code" | "rate_limited" | "no_session" | …`, and `resourceId` = user id.

`verify_totp_code` RPC writes a failure audit log entry directly when the code does not match — so even an app-side exception cannot hide a brute-force attempt. The RPC uses `pg_notify` would be overkill; instead it calls `public.log_audit_event` directly from inside the SECURITY DEFINER body.

## Error Handling

- **Invalid TOTP code shape:** `totpCodeSchema` rejects pre-RPC. Audit failure `reason: "invalid_input"`.
- **`verify_totp_code` returns false:** generic `"Invalid verification code"` to user. Audit failure `reason: "invalid_code"`. Increment rate-limit counter.
- **Recovery code mismatch / already used:** generic `"Invalid recovery code"`. Audit failure `reason: "invalid_code"` or `"already_used"`. Increment counter.
- **`verify_totp_code` RPC error (key missing, decrypt fails):** generic `"Verification failed. Please try again."`. Audit failure `reason: "rpc_error"`, `code: <pg_code>`.
- **Enrollment confirm without a pending secret:** generic error, audit `reason: "no_pending_secret"`.
- **Step-up cookie present but HMAC fails:** treat as absent. No audit (could be cookie-jar churn, not adversarial signal on its own). Force re-prompt.
- **Service-role JWT update fails after enrollment confirm:** rollback by calling `rpc('disable_totp')`; audit `auth.totp.enroll_complete` failure `reason: "metadata_update_failed"`; return generic error. User retries enrollment.
- **Disable: password re-prompt fails:** audit `auth.totp.disable` failure `reason: "password_invalid"`. No partial state change.

## Account Recovery Without Either Factor

If user loses authenticator AND recovery codes: account access is blocked behind 2FA. There is no in-band recovery from this state. Out-of-band recovery is operator-driven: support verifies identity via email + secondary factors (recent activity, payment proof, etc.) and runs `SELECT public.admin_disable_totp(<uid>)` against the service-role connection. That admin function is added with `REVOKE ALL FROM authenticated, anon` and only callable by `service_role`. Documented in `SECURITY.md` as a runbook; not exposed as a UI surface — by design, social-engineering this path requires a human in the loop.

## Testing

Unit (vitest, `tests/security/`):

- `totpCodeSchema` / `recoveryCodeSchema` — reject non-digit, wrong length, embedded whitespace edge cases.
- Cookie signer round-trip: sign(uid, exp) verifies; tamper any byte → invalid; wrong uid → invalid; past exp → invalid.
- `requireFreshTotp` — pass for non-enrolled users; pass with valid cookie; fail with missing cookie, expired cookie, tampered cookie.

Integration (vitest with mocked Supabase RPCs):

- Full enrollment flow: start → confirm → recovery-codes returned exactly once → second confirm without pending → error.
- Login challenge: middleware redirects on `app_metadata.totp_enabled=true`; correct code clears redirect; wrong code stays on challenge; rate-limited returns 429-shaped error.
- Step-up: createRideAction returns `2FA_STEPUP_REQUIRED` when enabled + no stepup cookie; passes through when cookie valid; passes through when 2FA not enabled.
- Recovery code single-use: same code accepted once then rejected.
- Disable: requires both password and TOTP code; clears app_metadata; clears recovery codes.

Security tests (`tests/security/`):

- Enumeration: `verify_totp_code` failure shape identical whether the user has TOTP enabled or not (modulo audit-event differences which are server-side, not observable to client).
- Brute force: 5 wrong codes within 15min returns 429; counter resets after window.
- Replay: `sb-2fa-passed` cookie from one user cannot be used by another (uid baked into HMAC).
- Step-up bypass: tampered `sb-2fa-stepup` cookie does not bypass `requireFreshTotp`.
- Recovery exhaustion: after 10 recovery codes used, user is locked out unless they have their authenticator.
- Plaintext leakage: snapshot test on `users` table reads (via test client) — `totp_secret_encrypted` column either absent from selected projection or returns null.

Manual (staging):

- End-to-end enrollment with Google Authenticator, Authy, 1Password, Bitwarden.
- Drift tolerance: change device clock ±25 seconds, verify passes; ±50 seconds, verify fails.
- Recovery code download UX: copy-to-clipboard, download .txt.
- Disable on a device that has lost its authenticator (uses recovery code, then triggers disable flow).

New env vars:

- `TOTP_COOKIE_SECRET` — 64+ random bytes (base64). Fail fast at module init if missing in production.
- (Existing) `SUPABASE_SERVICE_ROLE_KEY` — required for the `app_metadata` write at enrollment-complete and disable. Already present.

## Rollout

1. Run `SUPABASE_TOTP_2FA.sql` against the project. Verify Vault secret exists, RPC permissions correct.
2. Set `TOTP_COOKIE_SECRET` in Vercel env (production + preview).
3. Deploy code behind feature flag `NEXT_PUBLIC_TOTP_2FA_ENABLED` (default `false`). With flag off: settings UI hides the toggle, server actions return `"Feature disabled"`, middleware skips the redirect even if `app_metadata.totp_enabled === true` somehow (defense-in-depth).
4. Internal QA: enable flag for staff accounts via env override. Test full enroll → challenge → stepup → disable.
5. Flip flag in prod. Announce in product changelog.

Rollback: flip flag off. Users with active 2FA continue to authenticate by password only (because middleware short-circuits); their stored `totp_*` columns are untouched and will re-engage when the flag flips back on. No data migration to reverse.

## Out of Scope

- WebAuthn / passkeys as an alternative second factor — separate spec.
- SMS-based second factor — explicitly rejected (SIM-swap vector; the project already uses phone OTP for verification only, never authentication).
- Per-device "trust this browser for 30 days" — `sb-2fa-passed` is session-only by design.
- Enforcing 2FA org-wide (admin policy) — single-user opt-in only for v1.
- 2FA on the OAuth (Google) sign-in path — Google's account already has 2FA available; we do not re-prompt on top. If users opt into TOTP, the challenge still fires on every fresh login regardless of OAuth provider (the redirect is provider-agnostic — JWT flag check happens after session is set).
- Backup-code regeneration — out of v1. Users get 10 at enrollment; if depleted, they disable and re-enroll. Add `regenerateRecoveryCodesAction` in a follow-up.
- Step-up on the additional sensitive ops the user might think of later (account delete, email change, password change). v1 covers ride create and ride join (where contact phone is written). Trivial to add via `requireFreshTotp()`.

## Open Questions

None blocking. Two implementation-time choices left open:

- **QR code rendering library** — pure-client `qrcode` (npm) vs. inline SVG generator. Pick at implementation time; output format is a data: URL either way.
- **Recovery code display style** — grouped `XXXX-XXXX` 4-4 chars, or as a single 8-char string. Decided at implementation time after seeing the rendered UI.
